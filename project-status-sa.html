Schedule Assistant

Project Status — March 4, 2026 (v1.0)

Executive Summary

Schedule Assistant (SA) is a manager-only Electron desktop application that automates work order scheduling for field technicians. It reads work orders from SaaS (Salesforce), auto-assigns technicians via a 5-rule SLA engine using Haversine distance calculations, pushes the generated schedule to Technician Assistant (TA), and TA writes back to SaaS. The result is a full auto-sync pipeline — no manual scheduling needed.

The core problem it solves: Dispatchers currently spend hours each day manually assigning work orders to technicians, cross-referencing locations, travel distances, and technician availability. SA replaces this entire workflow with an automated engine that evaluates geographic proximity, travel constraints, daily job limits, and rolling zip code patterns to produce optimal schedules in seconds. Managers review the output on an 8-week calendar dashboard, and with one click the schedule flows downstream to every technician's TA instance and back to Salesforce.

How it works in practice:

  • Pull Queue: SA pulls queued (unassigned) work orders from SaaS into its local database. Each WO includes account name, address, coordinates, type, priority, and due date.
  • SLA Engine: The 5-rule engine runs against all queued WOs, evaluating each against all 8 associates. Rules are evaluated in priority order (R5 → R1 → R2 → R3 → R4), and the first matching rule assigns the WO to the optimal tech.
  • Schedule Push: Assigned WOs become ScheduleEntry records. The manager reviews on the 8-week calendar, then pushes to TA with one click.
  • TA Sync: TA receives the schedule entries and work orders, making them available to field technicians. When techs complete work, TA writes status back to SaaS.
  • Email Notifications: After scheduling, SA can send appointment confirmation emails to customers with the scheduled date, technician name, and work order details.
  • AI Chat: GPT-4.1-mini powered assistant with full visibility into all associates, the current schedule, and queued WOs. Managers can ask "who's free next Tuesday?" or "swap tech on WO-2026-015" and get schedule-aware suggestions.

Architecture summary: Same proven stack as Technician Assistant — Electron 33 desktop shell, Express 5 ESM backend, React 19 + Vite 7 dashboard, Prisma ORM with SQLite. Port 4400. Cookie-based auth (sa_auth). Mock adapter pattern (USE_MOCK=true) for all external integrations (SaaS, TA sync, email).

SLA Engine — 5 Rules

The SLA engine evaluates every queued work order against all active associates. Rules are checked in priority order. The first rule that matches assigns the WO. All distance calculations use the Haversine formula (great-circle distance between two lat/lng points on Earth).

Rule R5: Home-Area Assignment (Primary)

ConditionDetail
DistanceWO site is within 75 miles of the associate's home address
CapacityAssociate has fewer than 2 jobs scheduled on the target date
PriorityEvaluated first — local techs get local jobs before travel rules kick in
TiebreakerIf multiple techs qualify, the closest one wins
ActionCreates ScheduleEntry on WO due date, SlaLog with rule "R5", distance in miles

Rule R1: Closest Available Tech

ConditionDetail
DistanceClosest associate to WO site by Haversine (no distance cap)
CapacityAssociate has availability on the target date
FallbackRuns when R5 finds no home-area match (WO is outside all techs' 75mi zones)
ActionCreates ScheduleEntry, SlaLog with rule "R1", computed distance

Rule R2: Long-Distance Travel (>400mi)

ConditionDetail
DistanceClosest tech is more than 400 miles from WO site
Travel BufferSchedules WO for due date + 3 days to account for travel time
CapacityChecks availability on the travel-adjusted date
ActionCreates ScheduleEntry with adjusted date, SlaLog with rule "R2", notes travel buffer

Rule R3: On-Road Double-Up

ConditionDetail
TriggerAssociate already has a job scheduled in the area
DistanceNew WO is within 50 miles of an already-scheduled WO for the same tech on the same day
CapacityAssociate has not exceeded daily job limit
ActionStacks the WO on the same day as the existing job, SlaLog with rule "R3"

Rule R4: Rolling Zip Code

ConditionDetail
DistanceWO site is within 300 miles of the associate's current or recent job location
Time CutoffOnly considers jobs scheduled within the next 3 days (rolling window)
Cutoff Time3:00 PM — WOs arriving after 3 PM are evaluated for the next business day
ActionSchedules near the associate's travel path, SlaLog with rule "R4"

SLA Rule Evaluation Order

For each queued WO:
  1. R5: Check all associates within 75mi of home → assign closest with capacity
  2. R1: Find globally closest available tech → assign
  3. R2: If closest is >400mi, add 3-day travel buffer → assign on adjusted date
  4. R3: Check if any already-scheduled tech has a job within 50mi → stack same day
  5. R4: Check 300mi rolling zip from recent jobs within 3 days, 3PM cutoff → assign

Each assignment creates:
  - ScheduleEntry (workOrderId, associateId, scheduledDate)
  - SlaLog (woNumber, associateId, ruleApplied, distanceMi, reasoning)
  - WorkOrder.status updated to "scheduled"
  - WorkOrder.assignedTo set to associate ID

Auto-Sync Pipeline

The full pipeline runs as a chain of operations triggered by the manager clicking "Pull & Schedule" on the dashboard:

Step 1: SaaS → SA     (POST /api/sync/pull-queue)
  - Fetches unassigned WOs from Salesforce
  - Creates WorkOrder records with status "queued"
  - Logs: SyncLog { direction: "pull", recordCount: N }

Step 2: SA SLA Engine    (POST /api/sla/run)
  - Evaluates all queued WOs against 8 associates
  - Creates ScheduleEntry + SlaLog for each assignment
  - Updates WorkOrder.status to "scheduled"

Step 3: SA → TA         (POST /api/sync/push-schedule)
  - Pushes schedule entries to TA POST /api/sync/schedule
  - Pushes work order data to TA POST /api/sync/work-orders
  - TA auth bypassed on /api/sync paths
  - Logs: SyncLog { direction: "push-ta", recordCount: N }

Step 4: TA → SaaS       (triggered by TA after tech completes WO)
  - TA calls POST /api/sync/write-to-saas
  - Writes completion status back to Salesforce
  - Logs: SyncLog { direction: "push-saas" }

Step 5: SA → Email      (POST /api/email/appointment)
  - Sends appointment confirmations to customers
  - Includes: scheduled date, technician name, WO details
  - Logs: EmailLog { woNumber, recipientEmail, status }

Known Gotchas

  • Backend is ESM ("type": "module") — use dynamic import() not require(). backend-runner.js is CJS bridge.
  • Express 5 + new path-to-regexp: NO app.get("*") wildcards. Use app.use() middleware for SPA fallback.
  • SQLite does NOT support createMany with skipDuplicates. Seed uses findUnique + create loop or upsert instead.
  • Haversine distance returns statute miles. 1 degree latitude ~ 69 miles. Coordinates must be in decimal degrees (not DMS).
  • SLA engine processes WOs sequentially — earlier assignments affect capacity for later WOs. Order matters.
  • TA sync endpoints (/api/sync/*) bypass auth middleware. These paths are trusted internal communication between SA and TA.
  • Dashboard uses Vite 7. Rebuild dashboard BEFORE rebuilding Electron (dashboard/dist gets bundled).
  • Electron loads from http://localhost:4400 so session cookies work with SameSite=Lax.
  • .env is gitignored. Packaged app reads from ~/Library/Application Support/Schedule Assistant/.env.
  • Auto-migration in backend-runner.js via execFileSync of prisma db push (not migrate deploy).
