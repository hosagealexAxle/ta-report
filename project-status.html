Technician Assistant

Project Status — February 28, 2026 (v1.0 — Full Mock Build + AI Chat + Repair Docs)

Executive Summary

Technician Assistant (TA) is a field service management application that provides technicians with a controlled, read/write interface to Salesforce work order data — without giving them direct access to Salesforce or any other enterprise SaaS platform. The app is purpose-built for HVAC/mechanical field technicians who need to view assigned work orders, complete line items, upload site photos, track parts usage, report out-of-box issues, and manage their weekly schedule — all from a single mobile-friendly interface.

The core problem it solves: Field technicians currently need Salesforce licenses ($25–$150/user/month) to access work order data, but they only need a small subset of fields. TA acts as a controlled gateway — techs see exactly the fields they need (serial numbers, account info, line items, scheduling) and report their work (images, parts used, completion status) through purpose-built forms. The data syncs back to Salesforce through adapter APIs. This eliminates per-seat SaaS costs while giving dispatchers full visibility into field operations.

How it works in practice:

  • Login: Techs authenticate with username/password. 3-hour session cookies. Admin and tech roles with different permissions (only admin can change work order priority/severity).
  • Main Dashboard: Shows real-time weather for the tech's zip code, overdue work order alerts (auto-loaded on login), navigation to all app sections, and an AI chat assistant that can see all work orders, schedule data, and reference documents.
  • Work Orders: Browse all assigned WOs sorted by due date (oldest first). Each WO shows case info (serial number, account name/address, requested time), a priority meter (LOW/MED/HIGH — admin-only editing), and the work order type (Service, Install, Maintenance). Techs navigate between WOs with left/right arrows.
  • Type-Specific Templates: Each WO type renders a tailored workflow:
    • Service (Types 1, 4): 8 line items to check off, 4 image upload slots (system photos, damage, wide view), software push button, mandatory parts page before closing.
    • Install (Type 2): 5 line items, PDF/video installation guides, 10 image upload slots, out-of-box issue reporting, optional parts page.
    • Maintenance (Type 3): Same as Install but without OOB reporting.
  • Parts Page: 10-row parts form with quantity inputs, dropdown selection from 50-item parts catalog, confirmation checkboxes, 2 image uploads, FedEx return label generation for defective parts.
  • Out-of-Box Issues: Structured form for reporting shipping damage, system issues, or missing items with Y/N toggles, free-text notes, and 10 image upload slots.
  • Schedule: 8-week Monday–Sunday calendar view showing all scheduled WOs. Today is highlighted. Clicking a WO number navigates directly to that work order.
  • Notifications: Scrolling feed of up to 20 notifications (new assignments, priority changes, overdue alerts, parts shipments, training reminders) with color-coded urgency.
  • AI Chat: GPT-4.1-mini powered assistant with full visibility into all work orders, the complete schedule, and 6 repair reference PDFs. Techs can ask "what's overdue?", "what's on my schedule this week?", "give me the troubleshooting guide", or "what's diagnostic code E-42?" and get specific, data-backed answers with clickable document links.
  • Image Uploads: Base64 upload system with slot-based naming (system1, damage, install1–10, oob1–10, part1–2). Supports mobile camera picker. Thumbnails preview in-app. Stored locally in uploads/ directory per work order.

Mock Adapter Architecture: Since Salesforce, Equipment System, and FedEx APIs are not yet available, the entire app runs on a mock adapter layer that simulates real API behavior using seeded data in a local SQLite database. The mock layer is functionally identical to what the real adapters will provide — same data shapes, same response patterns, same error handling. When real APIs are available, each integration swaps by changing one environment variable (USE_MOCK=false) and implementing one adapter file per service. Zero frontend changes required.

Current state: The app is fully functional with 45 seeded work orders spanning 8 weeks, 50 parts catalog entries, 20 notifications, equipment data per WO, 6 repair reference PDFs, and a context-aware AI assistant. It runs as a macOS Electron desktop app accessible locally and via Tailscale on iPhone as a PWA.

BUILD COMPLETE (Feb 28, 2:30 AM): Full application built and deployed in a single session. 45 work orders across 8 weeks (3 overdue), 4 WO types with 3 template workflows, AI chat with full data context + 6 repair PDFs, admin-only priority control, custom shield icon, image uploads, parts management, FedEx label generation, and 8-week schedule calendar. Deployed to /Applications/Technician Assistant.app.

Application Function — Page-by-Page Detail

P1: Login Gate

  • SVG shield logo with gradient fill, metallic border, glow effect, and "TA" text
  • Username + password fields with Enter key submission
  • SHA-256 password hashing, server-side validation
  • 3-hour session cookie (ta_auth) with SameSite=Lax
  • Backend connectivity indicator while connecting
  • Two seeded accounts: admin / admin123 (full access) and tech1 / tech123 (read-only priority)

P2: Main Dashboard

  • Top bar: Mini shield logo, app name, current date (full weekday format), weather (temp + condition), MOCK badge, logout button
  • Overdue alerts: Red-bordered cards auto-loaded on login. Shows WO number, account name, due date. Click to navigate directly to WO.
  • Navigation grid: 5 buttons — Notifications (with unread count badge), Work Orders, Schedule, System Site (external link), LMS (external link)
  • AI Chat: 400px message area, text input with Enter-to-send, assistant responses with clickable PDF/URL links. System prompt includes all WOs, full schedule, and available repair documents.

P3: Notifications

  • Scrolling card list, max 20, newest first
  • Color-coded by type: green (info), orange (warning), red (urgent)
  • Left border accent color indicates severity
  • Shows title, body text, and relative date
  • Bottom bar with back-to-main + logout buttons

P4: Schedule (8-Week Calendar)

  • Monday–Sunday grid, 8 weeks scrollable
  • Today's cell highlighted with green accent border + background
  • WO numbers shown as clickable chips in each day cell
  • Click WO chip → navigates to that work order in WO hub
  • ~45 work orders distributed across 8 weeks (1–2 per weekday, Sundays empty)

P5–P13: Work Order Hub

  • WO Header: Left/right navigation arrows, WO number (large accent text), status badge (open/closed)
  • Case Info: 4-field grid — Serial Number, Account Name, Account Address, Time Requested
  • Priority Meter: 3-level bar (LOW green / MED yellow / HIGH red). Active level is full opacity, inactive are dimmed. Admin-only: tech users see read-only meter with "(admin only)" label. Server-side 403 enforcement.
  • Type Tabs: 4 tabs (1. Service, 2. Install, 3. Maintenance, 4. Service). Active tab highlighted with green accent.
  • Type-specific template renders below based on activeWO.type:
    • Type 1 or 4 → renderServiceTemplate()
    • Type 2 → renderInstallTemplate(showOob=true)
    • Type 3 → renderInstallTemplate(showOob=false)
  • System Data / Instructions: Split panel with equipment specs (capacity, voltage, phase, refrigerant, airflow, SEER, weight, model, firmware, install date, warranty, location) on left, PDF/video guide links on right.
  • Notes: Free-text textarea that saves per WO via PATCH endpoint.
  • Close button: Validates parts completion for service types before allowing close. Sets dateCompleted + timeCompleted automatically. Logs to ActionLog.

Known Gotchas

  • Backend is ESM ("type": "module") — use dynamic import() not require(). backend-runner.js is CJS bridge.
  • Express 5 + new path-to-regexp: NO app.get("*") wildcards. Use app.use() middleware for SPA fallback.
  • SQLite does NOT support createMany with skipDuplicates. Seed uses findUnique + create loop or upsert instead.
  • process.env is NOT available in the browser. External links (LMS, System Site) are served from the /api/health endpoint and stored in React state.
  • Electron loads from file:// — window.location.hostname is empty. API_BASE checks for empty string and falls back to localhost:4300.
  • Dashboard dist path: dev=../../dashboard/dist, packaged=process.resourcesPath/dashboard/dist. Server checks both.
  • Docs path: dev=../docs, packaged=process.resourcesPath/backend/docs. Server checks both.
  • .env is gitignored. Packaged app reads from ~/Library/Application Support/Technician Assistant/.env.
  • Auto-migration in backend-runner.js via execFileSync of prisma migrate deploy before server start.
  • Dashboard uses Vite 7. Rebuild dashboard BEFORE rebuilding Electron (dashboard/dist gets bundled).
  • macOS caches app icons aggressively. After icon changes, run lsregister -kill -r and killall Dock to force refresh.
  • Auth middleware bypasses localhost requests (for Electron). Remote (Tailscale) requests require valid session cookie.
  • Chat system prompt grows with WO count. At 45 WOs, prompt is ~4K tokens. Monitor if WO count grows significantly.
  • Image uploads are base64 encoded in JSON body (10MB limit). For production, switch to multipart form upload.
  • CORS is open (origin: "*") — Tailscale provides network-level security.
  • After changing PWA meta tags or manifest, user must delete and re-add PWA on iPhone home screen.
