Project Report — March 4, 2026 (v1.7)

1. Project Overview

A mobile-first field technician app that provides a controlled, secure interface to Salesforce data. Technicians never access Salesforce or any SaaS platform directly — instead they interact with specific read/write fields through this app, reducing friction and ensuring clean, safe data transfer back to the home office.

Core Goals

  • Controlled Access — Techs only see and touch the fields they need. No SaaS logins.
  • Real-Time Reporting — Work orders, parts, images, and notes sync back to Salesforce instantly.
  • Frictionless UX — Click-heavy (not type-heavy). Dropdowns, Y/N toggles, image uploads, QR scanning, pre-filled data.
  • AI Assistant — Built-in chat for looking up old work orders, getting help, and quick answers.
  • Parts Tracking — Local inventory management with QR scanning for parts in/out and manufacturer tracking.

6. Security & Access Control

  • No direct SaaS access — Techs never see Salesforce or any SaaS login
  • Password gate — 3-hour session timeout, re-auth required
  • Controlled fields only — Techs can only read/write specific fields per page
  • Read-only where needed — Case info, serial numbers, account details are read-only
  • Write only where needed — Completion Y/N, time completed, notes, image uploads
  • Admin-only controls — Priority level changes restricted to admin role
  • No data export — Data flows through controlled API endpoints only
  • QR scan audit trail — Every QR scan (in/out) is logged with timestamp, user, and reason

Security Hardening v1.3.1

  • bcrypt password hashing — Salted, 10 rounds via bcryptjs (replaced SHA-256)
  • HttpOnly session cookie — Session token cannot be read by client-side JavaScript
  • Login rate limiting — 5 attempts per minute per IP via express-rate-limit
  • CORS restricted — Only localhost, 127.0.0.1, and Tailscale IPs allowed (was origin: *)
  • Security headers — helmet middleware: X-Content-Type-Options, X-Frame-Options, Referrer-Policy
  • Auth bypass fix — Localhost detection uses req.ip (actual remote address) not req.hostname (client-spoofable)
  • Generic error responses — API errors return "server_error" instead of leaking internal error messages
  • Zip code validation — Weather endpoint validates /^\d{5}$/ before making external request

Security Hardening Phase 2 v1.5

  • Localhost auth bypass removed — All requests now require a valid session cookie, including localhost. No IP-based bypass.
  • Session IDs upgraded — Uses crypto.randomUUID() instead of Prisma cuid for cryptographically secure session tokens.
  • Secure cookie flag — Secure flag added to session cookie in production mode (HTTPS only).
  • Logout session invalidation — Logout now deletes the session record from the database (was only clearing the cookie).
  • Helmet CSP — Full Content Security Policy directives: default-src 'self', script-src 'self', style-src 'self' 'unsafe-inline', img-src 'self' data: blob:.
  • API rate limiter — 120 requests/minute across all /api/* routes.
  • Chat rate limiter — 10 requests/minute on /api/chat to control LLM costs.
  • Image upload validation — Magic byte checking (first 4 bytes) against known image signatures (JPEG, PNG, GIF, WebP, HEIC). Extension whitelist enforced.
  • Content-Type headers on image serve — Explicit MIME type set on all image serve endpoints with X-Content-Type-Options: nosniff.
  • Health endpoint hardened — Removed leaked internal URLs (lmsUrl, systemSiteUrl, partsOrderUrl).
  • Query parameter caps — take parameter capped at 100 on scan-log and action-log endpoints to prevent unbounded queries.
  • Wildcard CORS removed — Removed Access-Control-Allow-Origin: * from SPA fallback and static file serving.
  • resolveUserId hardened — No longer silently falls back to first tech user. Returns null if no authenticated user, triggering 401.
  • Strict CSP enforced — Removed inline debug script from dashboard HTML. scriptSrc: 'self' only (no unsafe-inline). crossOriginResourcePolicy: same-origin restored.
  • Electron loads via HTTP — Desktop app loads http://localhost:4300 instead of file:// so session cookies work correctly with SameSite=Lax.

QR Scan Feedback v1.1.1

After a successful scan, the camera closes and a toast notification slides up from the bottom of the screen showing the result (e.g. "QR IN: Compressor Module A-7 — qty now 3"). The toast auto-dismisses after 4 seconds. If the scan fails, a red error toast appears instead. The technician remains on the same page (inventory or parts) throughout.

iPhone Access v1.1.1

QR scanning requires HTTPS for camera access on iPhone Safari. The app is served over Tailscale HTTPS at https://alexs-mac-mini.tailb26e42.ts.net:8444. If something goes wrong during scanning, an error recovery screen appears with a Reload button instead of a blank page.

QR Swap Test Pair v1.2

The /api/test-qr-codes endpoint now includes 2 "SWAP TEST" entries at the end of the list. These simulate a real part replacement on a work order:

NEW Replacement Part: {"pn":"CMP-A007","name":"Compressor Module A-7","cat":"Compressor","sn":"CMP-NEW-20260301"}

OLD Failed Part: {"pn":"CMP-A007","name":"Compressor Module A-7","cat":"Compressor","sn":"CMP-OLD-20240815"}

Same part number, different serial numbers. Scan the OLD part with "Scan Removed" (enters reason), then scan the NEW part with "Scan Installed". Both actions log the part serial to the ActionLog.

Schedule Assistant Integration NEW v1.6

Schedule Assistant (SA) is a companion manager-only app that automates work order scheduling for TA's field technicians. SA runs on port 4400 and communicates with TA on port 4300 to complete a full automated data loop:

Data Flow

SaaS (Salesforce) → SA (pull WOs, 6-rule SLA auto-assign) → TA (receive schedule + WOs) → SaaS (write-back: queued → scheduled)

New TA Endpoints (v1.6)

POST /api/sync/schedule

Receives schedule entries from SA. Upserts by woNumber + scheduledDate. Creates new ScheduleEntry records or updates existing ones.

POST /api/sync/work-orders

Receives work order data from SA. Upserts by woNumber (unique). Creates new WorkOrder records or updates status/notes/scheduledDate.

POST /api/sync/write-to-saas

Writes WO status back to mock Salesforce, moving work orders from "in queue" to "scheduled". Closes the automation loop.

How It Works

  • SA's SLA engine auto-assigns the closest available tech using 6 rules based on Haversine distance from each tech's home zip.
  • New R6 (High Priority): Priority=3 WOs get emergency same-day dispatch. R6 runs FIRST before all other rules. If queued before noon AND within 50mi of a tech: same-day EVE slot (overrides 2-job limit). After noon or weekend: next business day. R6 calendar chips are red with pulse animation.
  • All /api/sync paths skip TA's auth middleware for internal SA-to-TA calls.
  • Techs see their auto-scheduled assignments on TA's Schedule page without any manual data entry.
  • Manager monitors the schedule in SA's 8-week calendar view with colored chips per WO type.

SA Security Hardening (12 Fixes) v1.7

Schedule Assistant received a 12-point security hardening pass: .env excluded from Electron bundle (C1), generic error responses on all 32 handlers (C2), auth middleware explicit allowlist (H1), path traversal protection on upload and serve (H2/H3), shell.openExternal restricted to http/https (H4), async bcrypt.compare (M1), SLA log bounded query (M2), body limit 256kb/12mb (M3), crypto.randomBytes(32) session IDs (L1), hourly session cleanup (L2), strict file extension whitelist (L3).
