5. SLA Engine

The core scheduling engine. Evaluates all queued WOs against all active associates using 6 priority-ordered rules with Haversine distance calculations.

POST /api/sla/run SLA ENGINE READ + WRITE

Frontend: App.jsx — runSlaEngine()
Backend: server.js — SLA engine
Executes the 6-rule SLA engine against all queued (unassigned) work orders. For each WO, evaluates all active associates in priority order (R6 → R5 → R1 → R2 → R3 → R4). R6 runs first for priority=3 WOs (emergency same-day dispatch). First matching rule assigns the WO. Creates ScheduleEntry + SlaLog for each assignment. Returns summary with total assigned, per-rule breakdown, and any unassigned WOs.
Reads: WorkOrder (findMany, status="queued"), Associate (findMany, active=true), ScheduleEntry (for capacity checks)
Writes: ScheduleEntry (create per assignment), SlaLog (create per assignment), WorkOrder (update status + assignedTo), ActionLog (create)

Haversine Distance Function

function haversineDistance(lat1, lng1, lat2, lng2) {
  const R = 3959; // Earth radius in statute miles
  const dLat = toRad(lat2 - lat1);
  const dLng = toRad(lng2 - lng1);
  const a = sin(dLat/2)^2 + cos(lat1) * cos(lat2) * sin(dLng/2)^2;
  const c = 2 * atan2(sqrt(a), sqrt(1-a));
  return R * c; // distance in miles
}

// In TA server.js auth middleware:

13. SLA Engine Integration Detail

How each of the 6 SLA rules maps to database operations. The engine processes all queued WOs sequentially — earlier assignments affect capacity for later WOs.

Rule R6: High Priority — Emergency Dispatch (RUNS FIRST)

R6 — High Priority (priority=3, same-day or next biz day) EMERGENCY

Runs FIRST before all other rules. For priority=3 WOs: if queued before noon local time AND a tech is within 50 miles of the WO site, assigns same-day dispatch using the EVE slot (overrides the normal 2-job-per-day limit). If queued after noon or on a weekend, schedules for next business day. Calendar chips for R6 entries render as red with a pulse animation.
Reads: Associate (all active, homeLat/homeLng), ScheduleEntry (for date check), WorkOrder (priority check)
Writes: ScheduleEntry (create, same-day or next biz day), SlaLog (ruleApplied="R6", distanceMi), WorkOrder (update status + assignedTo)

Rule R5: Home-Area Assignment

R5 — Home-Area (≤75mi, 2 jobs/day) PRIMARY RULE

For each queued WO, compute Haversine distance from WO site to every active associate's home coordinates. Filter to associates within 75 miles. Check ScheduleEntry count for target date < 2. Assign to closest qualifying associate.
Reads: Associate (all active, homeLat/homeLng), ScheduleEntry (count by associateId + date)
Writes: ScheduleEntry (create), SlaLog (ruleApplied="R5", distanceMi), WorkOrder (update status + assignedTo)

Rule R1: Closest Available Tech

R1 — Closest Tech (no distance cap) FALLBACK

When no associate is within 75mi (R5 fails), find the globally closest active associate with availability on the target date. No distance threshold — this is the catch-all assignment rule.
Reads: Associate (all active), ScheduleEntry (count by date)
Writes: ScheduleEntry (create), SlaLog (ruleApplied="R1", distanceMi), WorkOrder (update)

Rule R2: Long-Distance Travel

R2 — Long-Distance (>400mi, +3 day buffer) TRAVEL

When the closest tech is more than 400 miles from the WO site, the engine adds a 3-day travel buffer to the scheduled date. Checks availability on the adjusted date rather than the original due date.
Reads: Associate (closest), ScheduleEntry (count on adjusted date)
Writes: ScheduleEntry (create with adjusted date), SlaLog (ruleApplied="R2", notes travel buffer), WorkOrder (update)

Rule R3: On-Road Double-Up

R3 — On-Road Double (≤50mi from scheduled job) EFFICIENCY

Checks if any associate already has a job scheduled in the area. If a new WO is within 50 miles of an existing scheduled WO for the same tech on the same day, and the tech hasn't exceeded their daily max, stack the WO on that day.
Reads: ScheduleEntry (all with WO coordinates), WorkOrder (lat/lng for distance calc)
Writes: ScheduleEntry (create, same date), SlaLog (ruleApplied="R3"), WorkOrder (update)

Rule R4: Rolling Zip Code

R4 — Rolling Zip (≤300mi, 3-day window, 3 PM cutoff) GEOGRAPHIC

Looks at associates' recently scheduled jobs (within 3 days) and checks if the new WO is within 300 miles of any of those job sites. Applies a 3 PM cutoff — WOs arriving after 3 PM are evaluated for the next business day. Optimizes for geographic clustering along travel routes.
Reads: ScheduleEntry (recent 3 days with WO coordinates), WorkOrder (lat/lng)
Writes: ScheduleEntry (create), SlaLog (ruleApplied="R4"), WorkOrder (update)
