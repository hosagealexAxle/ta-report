Project Report — March 4, 2026 (v1.0)

1. Project Overview

Schedule Assistant is a manager-only Electron desktop app that automates work order scheduling for field technicians. It is the scheduling brain behind Technician Assistant — reading work orders from a SaaS queue, auto-assigning the closest available tech via a 5-rule SLA engine, writing the schedule back to TA, and notifying customers via appointment confirmation emails. The entire pipeline is automated: one button fires the full chain from SaaS pull to customer email.

Core Goals

  • Automated Scheduling — No manual tech assignment. The SLA engine picks the closest available technician using Haversine distance and 5 scheduling rules.
  • Full Pipeline Automation — SaaS pull, SLA assignment, TA push, SaaS write-back, and customer emails all fire in a single automated chain.
  • Manager Visibility — 8-week calendar grid, colored WO chips, collapsible dashboard with activity filters and stats.
  • AI Chat Assistant — Built-in GPT-powered chat with full schedule context for manager questions and tech swap suggestions.
  • Companion to TA — SA manages the schedule; TA executes it. SA pushes schedule entries and work orders directly to TA on port 4300.

4. SLA Engine — 5-Rule Scheduling System

The SLA engine is the core intelligence of Schedule Assistant. When new work orders arrive from SaaS, the engine evaluates each one against 5 rules in priority order to find the optimal technician assignment.

Rule 1 — Default

Closest Available Tech by Haversine Distance

The baseline rule. Calculate the great-circle distance between each active tech's home zip code and the job site address. Assign the closest available technician. This rule always runs as the fallback if no other rule matches first.

Rule 2 — Travel Days

Jobs >400mi Require Travel Day Before + After (3 Days Total)

If the Haversine distance from the tech's home zip to the job site exceeds 400 miles, the engine automatically books a travel day before the job and a travel day after. The job occupies 3 calendar days total: travel out, work day, travel home. Travel days appear as gray chips on the calendar.

Rule 3 — On-Road Double

Tech On-Site, Done Before Noon, 2nd Job Within 50mi, PM Slot

If a tech is already on-site for a morning job and is expected to finish before noon, the engine checks for another queued WO within 50 miles of the current job site. If found, it books the second job in the PM slot (12:00) on the same day. Maximizes daily output by doubling up when geographically convenient.

Rule 4 — Rolling Home Zip

Next-Day Job Within 300mi of Current Site, Decided Before 3pm, Skip Travel Home

If a tech finishes a job and there is a queued WO within 300 miles of their current location (not home zip), and the decision is made before 3:00 PM, the engine assigns the next-day job and the tech stays on the road instead of traveling home. The tech's "effective home zip" rolls forward to the current job site for the next calculation.

Rule 5 — Home-Area Double

Jobs Within 75mi of Home Zip, Book AM (9am) + PM (12pm), 2 Jobs/Day

When two or more queued WOs fall within 75 miles of a tech's home zip, the engine books both on the same day: first job at 9:00 AM, second job at 12:00 PM. This rule maximizes throughput for local-area work without requiring travel days.

Rule Priority: Rules are evaluated in order R2 → R3 → R4 → R5 → R1. The engine tries the most specific/optimization rules first and falls back to R1 (closest tech) if none of the special conditions are met. Every evaluation is logged to the SLA Engine Log (P8).

Distance Calculation

All distances use the Haversine formula — the great-circle distance between two points on a sphere given their latitudes and longitudes. Zip codes are geocoded to lat/lng coordinates. This provides accurate "as the crow flies" distance without requiring a routing API.

10. Security & Access Control

  • Manager-only access — No technician logins. Only managers can view and trigger the scheduling system.
  • Cookie-based auth — sa_auth cookie with 3-hour expiry. HttpOnly flag prevents client-side JavaScript access.
  • bcrypt password hashing — Salted, 10 rounds via bcryptjs
  • Login rate limiting — 5 attempts per minute per IP via express-rate-limit
  • CORS restricted — Only localhost, 127.0.0.1, and Tailscale IPs allowed
  • Security headers — helmet middleware: X-Content-Type-Options, X-Frame-Options, Referrer-Policy, CSP
  • Session IDs — Uses crypto.randomUUID() for cryptographically secure session tokens
  • Logout invalidation — Logout deletes the session record from the database
  • No data export — Data flows through controlled API endpoints only
  • Audit logging — Every SLA decision, sync operation, and email send is logged with timestamps

11. SA + TA Relationship

Schedule Assistant and Technician Assistant are two halves of the same system. SA handles the planning; TA handles the execution.

Schedule Assistant (SA)

Port 4400. Manager-only. Pulls WOs from SaaS, runs SLA engine, assigns techs, pushes schedule to TA, sends customer emails. Baby blue theme.

Technician Assistant (TA)

Port 4300. Tech-facing. Receives schedule from SA, displays WOs, collects field data (images, parts, notes), writes back to SaaS. Dark green theme.

Communication Flow

SA (:4400)
→
HTTP POST to :4300
→
TA (:4300)

SA pushes schedule entries and work order assignments to TA via direct HTTP POST requests on localhost. TA receives and stores the data, making it available to the field technician's interface.

Both apps must be running: SA pushes data to TA on port 4300. If TA is not running, the sync step will fail and be logged in the Sync Log for retry.
